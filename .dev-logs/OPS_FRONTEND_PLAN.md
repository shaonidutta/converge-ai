## ðŸ“Š DATABASE SCHEMA & BUSINESS LOGIC

### **Core Operations Tables:**

#### **1. Staff & RBAC Tables**

**`staff` Table:**
- Staff user accounts (separate from customer users)
- Fields: employee_id, role_id, first_name, last_name, email, mobile, password_hash
- Employment: department, designation, reporting_to, date_of_joining, date_of_leaving
- Security: login_attempts, locked_until, last_login
- Status: is_active, is_verified
- **Business Logic:** Self-referential hierarchy (reporting_to), account lockout after failed attempts

**`roles` Table:**
- Role definitions with hierarchical levels
- Fields: name, display_name, description, level (1-8), is_active
- **Business Logic:** Lower level = higher authority (Level 1 = Super Admin)

**`permissions` Table:**
- Granular permissions (module.action format)
- Fields: name, display_name, description, module, action
- Modules: bookings, users, providers, complaints, ops, system
- Actions: view, create, edit, delete, assign, resolve, export, etc.
- **Business Logic:** Permission-based authorization, not just role-based

**`role_permissions` Table:**
- Many-to-many junction table
- Maps roles to permissions
- **Business Logic:** Roles inherit permissions, staff inherit from roles

#### **2. Priority Queue Table**

**`priority_queue` Table:**
- AI-generated priority items for ops attention
- Fields: user_id, session_id, intent_type, confidence_score, priority_score, sentiment_score
- Intent Types: complaint, refund, cancellation, booking
- Review: is_reviewed, reviewed_by_staff_id, reviewed_at, action_taken
- **Business Logic:**
  - Priority score (0-100): AI-calculated based on intent confidence, sentiment, urgency
  - SLA risk calculation: Time since creation vs expected response time
  - Auto-escalation: Items not reviewed within SLA buffer get escalated
  - PII redaction: User details masked unless staff has full access permission

#### **3. Alert Tables**

**`alerts` Table:**
- System notifications for staff
- Fields: alert_type, severity, title, message, resource_type, resource_id
- Assignment: assigned_to_staff_id (NULL = broadcast to all)
- Status: is_read, is_dismissed, read_at, dismissed_at, expires_at
- Metadata: alert_metadata (JSON) for additional context
- **Business Logic:**
  - Auto-generated by alert rules
  - Broadcast alerts (NULL staff_id) visible to all
  - Expiring alerts auto-dismissed after expires_at

**`alert_rules` Table:**
- Configurable rules for automatic alert generation
- Fields: rule_name, rule_type, is_enabled, conditions (JSON), alert_config (JSON)
- Rule Types: sla, threshold, event
- **Business Logic:**
  - SLA rules: Check response_due_at and resolution_due_at
  - Threshold rules: Trigger when metric exceeds threshold
  - Event rules: Trigger on specific events (e.g., critical complaint created)

**`alert_subscriptions` Table:**
- Staff alert preferences
- Fields: staff_id, alert_type, is_enabled, delivery_channels (JSON)
- Delivery Channels: in_app, email, sms
- **Business Logic:** Staff can opt-in/out of specific alert types

#### **4. Audit & Config Tables**

**`ops_audit_log` Table:**
- Comprehensive audit trail for ops actions
- Fields: staff_id, action, resource_type, resource_id, pii_accessed
- Metadata: request_metadata (JSON), ip_address, user_agent
- **Business Logic:**
  - Log all PII access (expand=true requests)
  - Track who viewed/modified what and when
  - Compliance and security monitoring

**`ops_config` Table:**
- Runtime configuration (feature flags)
- Fields: config_key, config_value, description
- Default Configs:
  - `DEFAULT_STATUS_FILTER`: 'pending' (default filter for priority queue)
  - `SLA_BUFFER_HOURS`: '1' (hours before SLA breach to flag as at-risk)
  - `MAX_EXPAND_PER_HOUR`: '100' (rate limit for PII access)
  - `ENABLE_AUTO_ENRICHMENT`: 'true' (auto-fetch related entities)
- **Business Logic:** Dynamic configuration without code deployment

#### **5. Complaint & Booking Tables**

**`complaints` Table:**
- Customer complaints with SLA tracking
- Fields: booking_id, user_id, complaint_type, subject, description
- Priority: low, medium, high, critical (AI-assigned)
- Status: open, in_progress, resolved, closed, escalated
- Assignment: assigned_to_staff_id, assigned_at
- Resolution: resolved_by_staff_id, resolution, resolved_at
- SLA: response_due_at, resolution_due_at
- **Business Logic:**
  - SLA breach alerts auto-generated
  - Priority auto-assigned by AI based on sentiment and keywords
  - Auto-escalation if not resolved within SLA

**`bookings` Table:**
- Service bookings with payment tracking
- Fields: user_id, order_id, booking_number, payment_status, payment_method
- Amounts: subtotal, discount, GST (SGST/CGST/IGST), convenience_charge, total
- Status: pending, confirmed, in_progress, completed, cancelled
- Scheduling: preferred_date, preferred_time
- Settlement: is_settlement (pending/complete/failed/inprogress)
- **Business Logic:**
  - Payment status tracking
  - Partial payment support
  - Settlement tracking for provider payouts

## ðŸ” BUSINESS RULES & LOGIC

### **1. Authentication & Authorization**

**Authentication Flow:**
1. Staff logs in with email/mobile/employee_id + password
2. System validates credentials, checks is_active and is_verified
3. Increments login_attempts on failure, locks account after 5 attempts
4. Generates JWT access token (30 min) + refresh token (7 days)
5. Creates staff_session record with IP, user agent, expires_at
6. Returns tokens + staff profile + permissions list

**Authorization Flow:**
1. Every API request includes JWT token in Authorization header
2. Middleware validates token, extracts staff_id
3. Loads staff with role and permissions
4. Checks required permission for endpoint
5. Logs access in ops_audit_log if PII accessed
6. Returns 403 if permission denied

**Permission Hierarchy:**
- `system.admin`: Full system access (Super Admin only)
- `ops.*`: Operations module permissions
- `bookings.*`, `users.*`, `providers.*`, `complaints.*`: Entity permissions
- Granular actions: view, create, edit, delete, assign, resolve, export

### **2. Priority Queue Management**

**Priority Score Calculation (AI-based):**
```
priority_score = (
    intent_confidence * 40 +      # How confident AI is about intent (0-1)
    sentiment_urgency * 30 +       # Negative sentiment = higher urgency (0-1)
    time_decay * 20 +              # Older items get higher priority (0-1)
    user_history_factor * 10       # Repeat complainers get higher priority (0-1)
) * 100
```

**SLA Risk Calculation:**
```
sla_buffer_hours = config['SLA_BUFFER_HOURS']  # Default: 1 hour
time_since_creation = now - created_at
expected_response_time = 2 hours (for complaints), 4 hours (for refunds)

if time_since_creation > (expected_response_time - sla_buffer_hours):
    sla_risk = "at_risk"
elif time_since_creation > expected_response_time:
    sla_risk = "breached"
else:
    sla_risk = "on_track"
```

**PII Redaction Logic:**
```
if not staff.has_permission('ops.priority_queue.full_access'):
    user_mobile = mask_mobile(user_mobile)  # 98****5678
    user_email = mask_email(user_email)     # u***@example.com
    message_snippet = redact_pii(message_snippet)  # Remove phone, email, names
```

**Expand vs Summary:**
- **Summary (default):** Fast, returns only priority queue item + basic user info
- **Expand (expand=true):** Slower, fetches full related entity (complaint/booking/refund)
- **Rate Limiting:** Max 100 expand requests per staff per hour (configurable)
- **Audit Logging:** All expand requests logged with pii_accessed=true

### **3. Metrics Calculation**

**Booking Metrics:**
- Total bookings by status (pending, confirmed, in_progress, completed, cancelled)
- Today/week/month counts
- Growth rate: (current_month - previous_month) / previous_month * 100
- Revenue: Sum of total where payment_status='paid'

**Complaint Metrics:**
- Total complaints by priority (low, medium, high, critical)
- Total complaints by status (open, in_progress, resolved, closed, escalated)
- Unresolved count: status IN ('open', 'in_progress', 'escalated')
- Average resolution time: AVG(resolved_at - created_at) for resolved complaints

**SLA Metrics:**
- Total complaints with SLA tracking
- On-track: response_due_at > now AND resolution_due_at > now
- At-risk: response_due_at < (now + buffer) OR resolution_due_at < (now + buffer)
- Breached: response_due_at < now OR resolution_due_at < now
- Compliance rate: (on_track + at_risk) / total * 100

**Revenue Metrics:**
- Total revenue: SUM(total) where payment_status='paid'
- Today/week/month revenue
- Pending settlements: SUM(total) where is_settlement='pending'
- Refunded amount: SUM(total) where payment_status='refunded'

**Real-time Metrics:**
- Active sessions: COUNT(staff_sessions) where is_active=true
- Unread alerts: COUNT(alerts) where is_read=false
- Pending priority items: COUNT(priority_queue) where is_reviewed=false
- Critical complaints: COUNT(complaints) where priority='critical' AND status!='resolved'

### **4. Alert System Logic**

**Alert Rule Evaluation:**
1. Background job runs every 5 minutes
2. Evaluates all enabled alert rules
3. For each rule:
   - Check conditions (e.g., SLA breach, high workload)
   - If conditions met, generate alert
   - Assign to staff or broadcast based on alert_config
   - Send to delivery channels (in_app, email, sms)

**SLA Breach Alert Example:**
```json
{
  "rule_name": "sla_breach_critical",
  "rule_type": "sla",
  "conditions": {
    "priority": "CRITICAL",
    "buffer_hours": 1,
    "check_response": true,
    "check_resolution": true
  },
  "alert_config": {
    "alert_type": "sla_breach",
    "severity": "critical",
    "title_template": "CRITICAL: SLA Breach for Complaint #{resource_id}",
    "message_template": "Complaint #{resource_id} has breached SLA deadline. Immediate action required.",
    "assign_to_owner": true,
    "broadcast_to_managers": true
  }
}
```

**Alert Delivery:**
- **In-app:** Always delivered, shown in alert bell icon
- **Email:** Sent if staff subscribed to alert_type with 'email' channel
- **SMS:** Sent if staff subscribed to alert_type with 'sms' channel

### **5. Staff Management Logic**

**Role Hierarchy (Level 1 = Highest):**
1. **Super Admin (Level 1):** Full system access, all permissions
2. **Admin (Level 2):** All permissions except system.admin
3. **Manager (Level 3):** Team management, view all, edit most
4. **Senior Agent (Level 4):** Advanced operations, assign/resolve
5. **Agent (Level 5):** Standard operations, view/edit assigned items
6. **Support (Level 6):** Customer support, view/assign only
7. **Analyst (Level 7):** Data analysis, view/export metrics
8. **Viewer (Level 8):** Read-only access, no modifications

**Permission Inheritance:**
- Staff inherits all permissions from their role
- Permissions checked on every API request
- Can check single permission: `staff.has_permission('bookings.edit')`
- Can check any of multiple: `staff.has_any_permission(['bookings.edit', 'bookings.view'])`
- Can check all of multiple: `staff.has_all_permissions(['bookings.edit', 'bookings.delete'])`

**Account Security:**
- Password hashed with bcrypt
- Login attempts tracked, account locked after 5 failures
- Locked until timestamp set to now + 30 minutes
- Session tracking with IP and user agent
- Session expires after 7 days of inactivity
